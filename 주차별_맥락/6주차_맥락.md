## Handle 패턴 — relay에서 시작, 전 드라이버로 확장

### Handle 패턴이란?

C는 클래스/객체가 없으므로, 동일한 하드웨어가 여러 개일 때 **구조체 + 함수 첫 인자 포인터** 패턴으로 인스턴스를 구분한다. STM32 HAL 전체가 이 패턴을 사용한다(`HAL_ADC_Start(&hadc1)`, `HAL_I2C_Init(&hi2c1)` 등).

```
[기존 방식 — 전역 변수]

static GPIO_TypeDef *port_;      ← 파일 전역, 1개만 가능
static uint16_t pin_;
static Relay_State relay_state_;

void Relay_Init(GPIO_TypeDef *port, uint16_t pin);
void Relay_Set(Relay_State state);       ← 어느 릴레이인지 구분 불가
Relay_State Relay_Get(void);
```

```
[핸들 패턴 — 구조체 인스턴스]

typedef struct {
    GPIO_TypeDef *port;
    uint16_t      pin;
    Relay_State   state;         ← 인스턴스마다 독립
} Relay_Handle;

void Relay_Init(Relay_Handle *relay, GPIO_TypeDef *port, uint16_t pin);
void Relay_Set(Relay_Handle *relay, Relay_State state);   ← 어느 릴레이인지 명확
Relay_State Relay_Get(Relay_Handle *relay);
```

릴레이가 2개, 3개 추가되어도 코드 복사 없이 인스턴스만 늘리면 된다:

```c
Relay_Handle pump_relay;    // 펌프용
Relay_Handle light_relay;   // 조명용 (나중에 추가)

Relay_Init(&pump_relay, GPIOB, GPIO_PIN_0);
Relay_Init(&light_relay, GPIOC, GPIO_PIN_5);
```

Kotlin과 비교하면:

| 개념 | Kotlin | C (핸들 패턴) |
|------|--------|--------------|
| 인스턴스 | `val relay = Relay(port, pin)` | `Relay_Handle h; Relay_Init(&h, port, pin)` |
| 멤버 변수 | `this.port` | `relay->port` |
| 메서드 호출 | `relay.set(ON)` | `Relay_Set(&relay, ON)` |

### Data 구조체를 Handle 안에 포함한 이유

처음엔 `RHT01_Data data;`를 Handle 밖에 별도 변수로 둘지 고민했다. 결론은 **Handle 안에 포함**하는 것이 맞다.

```c
// ❌ 외부 변수 방식 — 센서가 2개가 되면?
RHT01_Handle rht1, rht2;
RHT01_Data   rht1_data, rht2_data;   // 짝을 맞춰 관리해야 함

// ✅ Handle 내부 포함 — 센서 추가 시 Handle 하나만 추가
RHT01_Handle rht1, rht2;             // 데이터는 각 Handle 안에 자동으로 독립
```

Handle 하나로 해당 하드웨어의 **설정(port, pin, hadc…) + 현재 상태(data)** 를 모두 캡슐화한다.

## 최종 구조체 설계

### RHT01_Handle

```c
typedef struct {
    float humidity;
    float temperature;
} RHT01_Data;

typedef struct {
    GPIO_TypeDef *port;   // 연결된 GPIO 포트
    uint16_t      pin;    // 연결된 GPIO 핀
    RHT01_Data    data;   // 마지막 측정 결과
} RHT01_Handle;

void              RHT01_Init(RHT01_Handle *handle, GPIO_TypeDef *port, uint16_t pin);
HAL_StatusTypeDef RHT01_Read(RHT01_Handle *handle);
HAL_StatusTypeDef RHT01_ReadMedian(RHT01_Handle *handle);
```

### SoilSensor_Handle

```c
typedef struct {
    uint32_t adc_value;   // 원시 ADC 값 (0~4095)
    float    voltage;     // 전압 (0.0~3.3V)
} SoilSensor_Data;

typedef struct {
    ADC_HandleTypeDef *hadc;
    SoilSensor_Data    data;
    uint16_t adc_dry;    /**< 건조할 때 ADC raw value (보정값) */
    uint16_t adc_wet;    /**< 완전 젖었을 때 ADC raw value (보정값) */
} SoilSensor_Handle;

HAL_StatusTypeDef SoilSensor_Init(SoilSensor_Handle *handle, ADC_HandleTypeDef *hadc, uint16_t adc_dry, uint16_t adc_wet);
HAL_StatusTypeDef SoilSensor_Read(SoilSensor_Handle *handle);
```

`adc_dry`, `adc_wet`은 센서마다 다를 수 있는 **보정(캘리브레이션) 값**이다. 헤더에 기본값 상수도 제공한다:

```c
#define SOIL_SENSOR_ADC_DRY_DEFAULT  4090
#define SOIL_SENSOR_ADC_WET_DEFAULT  1700
```

### SSD1306_Handle

```c
typedef struct {
    I2C_HandleTypeDef *hi2c;
    uint8_t  i2c_addr;
    uint8_t  framebuffer[1024];   // 128×64 / 8 = 1024바이트
} SSD1306_Handle;

HAL_StatusTypeDef SSD1306_Init(SSD1306_Handle *handle, I2C_HandleTypeDef *hi2c, uint8_t addr);
void SSD1306_Clear(SSD1306_Handle *handle, SSD1306_Color color);
HAL_StatusTypeDef SSD1306_Update(SSD1306_Handle *handle);
void SSD1306_DrawPixel(SSD1306_Handle *handle, uint8_t x, uint8_t y, SSD1306_Color color);
void SSD1306_WriteChar(SSD1306_Handle *handle, uint8_t x, uint8_t page, char ch);
void SSD1306_WriteString(SSD1306_Handle *handle, uint8_t x, uint8_t page, const char *str);
```

프레임버퍼를 Handle 안에 배열로 직접 포함시켰다. 포인터가 아닌 배열이므로 Handle 선언 시 자동으로 1024바이트가 할당된다.

```c
#define SSD1306_WIDTH       128
#define SSD1306_HEIGHT      64
#define SSD1306_PAGE_PIXEL  8     // SSD1306 컨트롤러의 하드웨어 상수 (문자와 무관)
```

## SSD1306 페이지 아키텍처

`SSD1306_PAGE_PIXEL 8`은 "문자 높이 8픽셀"이 아니라, **SSD1306 컨트롤러의 하드웨어 메모리 구조**에서 비롯된 상수다. SSD1306은 화면을 **페이지(page)** 단위로 나누어 관리한다.

```
[SSD1306 메모리 구조 — 128×64]

Page 0 : 행 0~7    (세로 8픽셀)
Page 1 : 행 8~15
Page 2 : 행 16~23
Page 3 : 행 24~31
Page 4 : 행 32~39
Page 5 : 행 40~47
Page 6 : 행 48~55
Page 7 : 행 56~63
         ← 128열 →
```

각 페이지의 각 열은 1바이트(8비트)로 세로 8픽셀을 표현한다. 따라서 전체 프레임버퍼 = 128열 × 8페이지 = **1024바이트**가 된다.

이 8이라는 숫자는 디스플레이 크기와 무관하게 SSD1306 컨트롤러라면 항상 8이다. 128×32 디스플레이도 페이지당 8픽셀로 동일하다. 우리 프로젝트에서 5×8 폰트를 쓰는 것은 이 하드웨어 페이지 높이(8픽셀)에 맞춘 것이지, 페이지 구조가 글자 때문에 생긴 것이 아니다.

## `#include "main.h"` vs 직접 HAL 헤더 include

- `main.h`를 include하면 편리하지만, 해당 BSP가 **특정 프로젝트의 CubeMX 설정에 의존**하게 된다.
- 핸들 패턴처럼 핀 매크로(`WATER_PUMP_Pin` 등)를 사용하지 않는 드라이버는 필요한 HAL 헤더만 직접 include하면 **다른 프로젝트에서도 재사용** 가능하다.

## HAL 순환 include 버그 — 근본 원인과 해결책

### 오류 증상

빌드 시 거의 모든 파일에서:
```
error: unknown type name 'HAL_StatusTypeDef'
error: unknown type name 'HAL_LockTypeDef'
```

### 순환 include가 발생하는 경위

개별 HAL 헤더(`stm32f1xx_hal_def.h`, `stm32f1xx_hal_gpio.h`, `stm32f1xx_hal_adc.h` 등)를 직접 include하면 다음 경로로 순환이 생긴다:

```
내 파일
  └─ #include "stm32f1xx_hal_def.h"   ← 인클루드 가드 설정됨
       └─ #include "stm32f1xx.h"
            └─ (USE_HAL_DRIVER 플래그 감지)
                 └─ #include "stm32f1xx_hal.h"
                      └─ #include "stm32f1xx_hal_def.h"  ← 가드에 의해 SKIP!
                      └─ #include "stm32f1xx_hal_conf.h"
                           └─ #include "stm32f1xx_hal_rcc.h"
                                 └─ HAL_StatusTypeDef 사용 → 아직 미정의! → 오류
```

`stm32f1xx_hal_def.h`가 자기 가드를 설정한 뒤 안에서 `stm32f1xx.h`를 include하고, 그 안에서 다시 `stm32f1xx_hal_def.h`를 포함하려 하지만 가드에 막혀 건너뛰어 버린다. 이 때문에 `HAL_StatusTypeDef`가 정의되기 전에 다른 헤더들이 그것을 사용하게 된다.

### 해결책

**모든 BSP 파일에서 HAL의 진입점은 반드시 `stm32f1xx_hal.h` 하나로 통일한다.**

```c
// ❌ 잘못된 방식
#include "stm32f103xb.h"
#include "stm32f1xx_hal_def.h"
#include "stm32f1xx_hal_adc.h"
#include "stm32f1xx_hal_i2c.h"
#include "stm32f1xx_hal_gpio.h"

// ✅ 올바른 방식
#include "stm32f1xx_hal.h"   // 이 하나로 HAL 전체가 올바른 순서로 포함됨
```

`stm32f1xx_hal.h`는 자신의 가드를 설정한 *뒤에* 체인을 시작하므로, 순환이 발생하지 않는다.

## 이번 주 배운 것들

### 1. GPIO 초기값(Output Level)의 중요성

CubeMX에서 GPIO Output 핀을 설정할 때 **초기값(Output Level)**을 반드시 고려해야 한다. MCU 리셋 직후 핀이 잠깐이라도 의도치 않은 레벨로 튀면, 릴레이가 순간적으로 동작하여 펌프가 켜질 수 있다.

- HIGH-active 릴레이 → 초기값 `Low` (OFF 상태 유지)
- LOW-active 릴레이 → 초기값 `High` (OFF 상태 유지)

### 2. 릴레이 Active 방향은 실측이 기준

데이터시트나 아키텍처 문서에 "LOW-active"로 기재되어 있더라도, 실제 모듈의 PCB 회로 설계에 따라 달라진다. JQC-3FF-S-Z 릴레이 자체는 코일에 전압이 걸리면 동작하는 단순한 부품이고, **모듈 PCB의 구동 회로**가 active 방향을 결정한다.

이 프로젝트에서는 실측 결과 HIGH-active로 확인되었고, 드라이버를 그에 맞게 구현했다.

#### 릴레이 모듈의 구동 회로 — 트랜지스터 vs 옵토커플러

릴레이 코일은 수십 mA의 전류가 필요하지만, MCU GPIO는 보통 수~20mA 정도만 공급할 수 있다. 따라서 릴레이 모듈에는 GPIO 신호를 증폭하여 코일을 구동하는 회로가 반드시 들어간다.

**① NPN 트랜지스터 방식 (HIGH-active)**

```
GPIO (High) ──저항──▶ 베이스(B)
                       │
                   NPN 트랜지스터
                       │
               콜렉터(C) ── 릴레이 코일 ── VCC
                       │
               에미터(E) ── GND
```

- GPIO High → 베이스에 전류 유입 → 트랜지스터 ON → 콜렉터-에미터 도통 → 코일에 전류 → 릴레이 동작
- 구조가 단순하고 저렴.

**② 옵토커플러 방식 (대부분 LOW-active)**

옵토커플러(Optocoupler)는 **LED + 포토트랜지스터**가 하나의 패키지 안에 들어 있는 부품이다. 입력 쪽 LED가 빛을 내면, 출력 쪽 포토트랜지스터가 그 빛을 감지하여 ON된다. 전기적으로는 완전히 분리(광 절연, Galvanic Isolation)되어 있다.

```
옵토커플러 내부 구조 (예: PC817)

┌────────────────────────────┐
│  입력 쪽        출력 쪽     │
│                            │
│  1(A) ──▶│──  4(C)        │
│        LED  ~~~▶  포토TR   │  ~~~ = 빛 (광 결합)
│  2(K) ──▶│──  3(E)        │
│                            │
└────────────────────────────┘
핀: 1=Anode, 2=Cathode, 3=Emitter, 4=Collector
```

```
[입력 쪽 — MCU와 연결]
VCC(3.3V) ──저항──▶ 1(A) LED 2(K) ◀── GPIO

[출력 쪽 — 릴레이 코일 구동]
VCC(5V) ── 릴레이 코일 ── 4(C) 포토TR 3(E) ── GND
```

- GPIO Low → LED 양단에 전위차 발생 → LED 발광 → 포토트랜지스터 ON → 코일에 전류 → 릴레이 동작
- GPIO High → LED 양단 전위차 없음 → LED 꺼짐 → 릴레이 해제

**광 절연의 장점:** MCU 쪽과 릴레이/부하 쪽이 전기적으로 완전히 분리됨. 역기전력(Back-EMF)이나 고전압 부하의 노이즈가 MCU로 전달되지 않음. 산업용, 고전압(AC 220V 등) 환경에서 MCU 보호에 필수적.

| 방식 | Active 방향 | 특징 |
|------|------------|------|
| NPN 트랜지스터 | HIGH-active | 단순, 저렴, MCU-릴레이 전기적 연결됨 |
| 옵토커플러 | 대부분 LOW-active | 광 절연으로 MCU 보호, 고전압 환경에 적합 |

> 모듈 PCB를 보면 구분할 수 있다: 검은색 4핀 칩(옵토커플러, 예: PC817)이 보이면 옵토커플러 방식, 3핀 트랜지스터(예: S8050)만 보이면 트랜지스터 방식이다.

### 3. C에서의 인스턴스화와 Init 함수 패턴

C에서는 구조체 변수를 선언할 때 "인스턴스화(instantiation)"라고 한다. `new` 키워드가 없고, 변수를 선언하는 것 자체가 인스턴스를 만드는 행위다.

```c
// Kotlin: val soil = SoilSensor(hadc, 4090, 1700)
// C:
SoilSensor_Handle soil_handle;                           // 인스턴스화 (선언)
SoilSensor_Init(&soil_handle, &hadc1, 4090, 1700);       // 초기화
```

임베디드에서 `Init()` 함수는 두 가지를 한다:
1. **구조체 필드 채우기** — `handle->hadc = hadc;`
2. **하드웨어 초기화** — `HAL_ADCEx_Calibration_Start()`, SSD1306 init sequence 등

**C99 지정 초기화(Designated Initializer):** 선언 시 특정 필드만 초기화할 수 있다:

```c
static SoilSensor_Handle soil_handle = {
    .adc_dry = SOIL_SENSOR_ADC_DRY_DEFAULT,
    .adc_wet = SOIL_SENSOR_ADC_WET_DEFAULT,
};
```
명시하지 않은 필드는 0으로 초기화된다. 하지만 HAL 핸들(`hadc`, `hi2c` 등)은 런타임에야 알 수 있으므로, 보통 `Init()` 함수로 초기화한다.

### 4. static 함수와 순방향 선언(Forward Declaration)

`.c` 파일 내부에서만 쓰는 함수는 `static`으로 선언하여 외부에서 접근할 수 없게 한다. 구현이 호출 위치보다 아래에 있으면 컴파일러가 오류를 내므로, **순방향 선언**으로 해결한다.

```c
// rht01.c 상단 — 선언만 (세미콜론으로 끝남)
static void Set_Output(GPIO_TypeDef *port, uint16_t pin);
static void Set_Input(GPIO_TypeDef *port, uint16_t pin);
static void delay_us(uint32_t us);

// 공개 함수 (위에서 선언된 함수들을 자유롭게 호출 가능)
HAL_StatusTypeDef RHT01_Read(RHT01_Handle *handle) {
    Set_Output(handle->port, handle->pin);  // 문제없음
    ...
}

// 파일 하단 — 실제 구현
static void Set_Output(GPIO_TypeDef *port, uint16_t pin) { ... }
```

**Doxygen 주석 위치:** 선언부(상단)가 아닌 **구현부(하단)** 에 단다. Doxygen이란 C/C++ 코드에서 `/** ... */` 형식의 주석을 파싱해 자동으로 API 문서를 생성하는 도구다.

### 5. Doxygen 구조체 멤버 인라인 주석

```c
typedef struct {
    ADC_HandleTypeDef *hadc;
    SoilSensor_Data    data;
    uint16_t adc_dry;    /**< 건조할 때 ADC raw value */
    uint16_t adc_wet;    /**< 완전 젖었을 때 ADC raw value */
} SoilSensor_Handle;
```

`/**<` 에서 `<`가 핵심이다. `<`가 없으면 **다음 항목**의 주석, `<`가 있으면 **바로 앞(왼쪽) 항목**의 주석으로 인식한다.

### 6. RHT01_ReadMedian에서 float 배열 복사

`RHT01_ReadMedian()`은 정확도를 높이기 위해 여러 번 측정한 중앙값을 사용한다. 이때 측정값을 임시 배열에 수집해야 하는데, `RHT01_Read(handle)`를 호출할 때마다 `handle->data`가 **덮어쓰인다**. 이전 측정값을 보존하려면 반드시 값을 복사해 두어야 한다.

```c
float hum[5], temp[5];
...
hum[valid]  = handle->data.humidity;     // ✅ 값 복사 (float 값 자체가 복사됨)
temp[valid] = handle->data.temperature;  // ✅ 값 복사
```

포인터 배열 `float *hum[5]`로 만들면 같은 주소를 저장하게 되어, 다음 Read()에서 모두 덮어쓰인다. `float hum[5]`는 float 값을 저장하는 배열이므로 항상 값 복사가 일어난다.

### 7. static 지역 변수와 BSS 영역

`ssd1306.c`의 `SSD1306_Update()` 내부:

```c
HAL_StatusTypeDef SSD1306_Update(SSD1306_Handle *handle) {
    static uint8_t data_buf[1 + SSD1306_WIDTH * SSD1306_HEIGHT / SSD1306_PAGE_PIXEL];
    //     ^^^^^^ static!
    ...
}
```

`data_buf`는 1025바이트다. `static` 없이 선언하면 이 함수가 호출될 때마다 **스택에 1025바이트를 할당했다 해제**하는 오버헤드가 생긴다.

`static`을 붙이면 프로그램 시작 시 **BSS 영역**에 1회 할당된다. 임베디드에서 큰 배열을 함수 내부에 둘 때는 `static`을 붙여 스택 사용을 줄이는 것이 일반적이다. STM32F103RB의 SRAM은 20KB뿐이다.

**메모리 영역 정리:**

| 영역 | 내용 |
|------|------|
| Flash | 코드, const 데이터 (ROM) |
| BSS | 초기값 없는 전역/static 변수 (0으로 초기화) |
| Data | 초기값 있는 전역/static 변수 |
| Stack | 지역 변수, 함수 호출 시 자동 할당/해제 |
| Heap | 동적 할당 (임베디드에서 보통 사용 안 함) |

### 8. 발견한 버그: `&handle->port` vs `handle->port`

```c
// ❌ 잘못된 코드
GPIO_TypeDef *port = &handle->port;
// handle->port는 이미 GPIO_TypeDef* (포인터)
// 여기에 &를 붙이면 포인터의 주소 = GPIO_TypeDef** 가 됨

// ✅ 올바른 코드
GPIO_TypeDef *port = handle->port;
```

### 9. SPI1_NSS 등 핀 겸용 기능(Alternate Function) 확인

STM32 핀은 대부분 여러 기능을 겸용한다. GPIO로 사용하려는 핀이 SPI, I2C, UART 등의 Alternate Function과 겹치는지 확인해야 한다. PA4는 SPI1_NSS 겸용이므로, 나중에 SPI 장치를 추가할 가능성을 고려하면 피하는 것이 안전하다. 데이터시트의 Alternate Function Mapping 테이블에서 확인 가능하다.
