## 릴레이 BSP 드라이버 설계

### 핸들 패턴 (Handle Pattern)

릴레이를 나중에 추가하더라도 같은 드라이버를 재사용할 수 있도록, STM32 HAL과 동일한 **핸들 패턴**을 적용했다.

핸들 패턴이란: 각 인스턴스(릴레이 1개)를 **구조체로 만들고**, 모든 함수에 **포인터로 넘기는 방식**이다. HAL의 `ADC_HandleTypeDef`, `I2C_HandleTypeDef`가 같은 패턴이다.

```
[기존 방식 — 전역 변수]

static GPIO_TypeDef *port_;      ← 파일 전역, 1개만 가능
static uint16_t pin_;
static Relay_State relay_state_;

void Relay_Init(GPIO_TypeDef *port, uint16_t pin);
void Relay_Set(Relay_State state);       ← 어느 릴레이인지 구분 불가
Relay_State Relay_Get(void);
```

```
[핸들 패턴 — 구조체 인스턴스]

typedef struct {
    GPIO_TypeDef *port;
    uint16_t      pin;
    Relay_State   state;         ← 인스턴스마다 독립
} Relay_Handle;

void Relay_Init(Relay_Handle *relay, GPIO_TypeDef *port, uint16_t pin);
void Relay_Set(Relay_Handle *relay, Relay_State state);   ← 어느 릴레이인지 명확
Relay_State Relay_Get(Relay_Handle *relay);
```

이렇게 하면 릴레이가 2개, 3개 추가되어도 코드 복사 없이 인스턴스만 늘리면 된다:

```c
Relay_Handle pump_relay;    // 펌프용
Relay_Handle light_relay;   // 조명용 (나중에 추가)

Relay_Init(&pump_relay, GPIOB, GPIO_PIN_0);
Relay_Init(&light_relay, GPIOC, GPIO_PIN_5);
```

### `#include "main.h"` 대신 직접 HAL 헤더 사용

기존 BSP 드라이버들(`soil_sensor.h`, `rht01.h`)은 `#include "main.h"`를 사용했다. `main.h`는 아래 두 가지를 한 번에 가져오는 편의용 헤더다:

1. `stm32f1xx_hal.h` → HAL 타입과 함수 (`HAL_GPIO_WritePin`, `GPIO_PIN_SET` 등)
2. CubeMX가 생성한 핀 매크로 (`RELAY_GPIO_Port`, `RELAY_Pin` 등)

릴레이 드라이버에서는 핸들 패턴을 사용하므로 CubeMX 매크로(`WATER_PUMP_Pin` 등)에 의존하지 않는다. 따라서 `main.h` 대신 필요한 HAL 헤더만 직접 include했다:

- `relay.h` → `#include "stm32f103xb.h"` (GPIO_TypeDef 정의)
- `relay.c` → `#include "stm32f1xx_hal_gpio.h"` (HAL_GPIO_WritePin 함수)

### 파일 구조

**`BSP/Inc/relay.h`:**
- `Relay_State` enum — `RELAY_OFF(0)`, `RELAY_ON(1)`
- `Relay_Handle` struct — `port`, `pin`, `state`
- 함수 선언 3개: `Relay_Init`, `Relay_Set`, `Relay_Get`

**`BSP/Src/relay.c`:**
- `Relay_Init()` — port/pin을 구조체에 저장, GPIO Low 출력(OFF 확정)
- `Relay_Set()` — HIGH-active 처리 (ON→GPIO_PIN_SET, OFF→GPIO_PIN_RESET), state 갱신
- `Relay_Get()` — 구조체의 state 필드 반환

## 이번 주 배운 것들

### 1. GPIO 초기값(Output Level)의 중요성

CubeMX에서 GPIO Output 핀을 설정할 때 **초기값(Output Level)**을 반드시 고려해야 한다. MCU 리셋 직후 핀이 잠깐이라도 의도치 않은 레벨로 튀면, 릴레이가 순간적으로 동작하여 펌프가 켜질 수 있다.

- HIGH-active 릴레이 → 초기값 `Low` (OFF 상태 유지)
- LOW-active 릴레이 → 초기값 `High` (OFF 상태 유지)

### 2. 릴레이 Active 방향은 실측이 기준

데이터시트나 아키텍처 문서에 "LOW-active"로 기재되어 있더라도, 실제 모듈의 PCB 회로 설계에 따라 달라진다. JQC-3FF-S-Z 릴레이 자체는 코일에 전압이 걸리면 동작하는 단순한 부품이고, **모듈 PCB의 구동 회로**가 active 방향을 결정한다.

이 프로젝트에서는 실측 결과 HIGH-active로 확인되었고, 드라이버를 그에 맞게 구현했다.

#### 릴레이 모듈의 구동 회로 — 트랜지스터 vs 옵토커플러

릴레이 코일은 수십 mA의 전류가 필요하지만, MCU GPIO는 보통 수~20mA 정도만 공급할 수 있다. 따라서 릴레이 모듈에는 GPIO 신호를 증폭하여 코일을 구동하는 회로가 반드시 들어간다. 대표적으로 두 가지 방식이 있다.

**① NPN 트랜지스터 방식 (HIGH-active)**

```
GPIO (High) ──저항──▶ 베이스(B)
                       │
                   NPN 트랜지스터
                       │
               콜렉터(C) ── 릴레이 코일 ── VCC
                       │
               에미터(E) ── GND
```

- GPIO High → 베이스에 전류 유입 → 트랜지스터 ON → 콜렉터-에미터 도통 → 코일에 전류 흐름 → 릴레이 동작
- GPIO Low → 트랜지스터 OFF → 릴레이 해제
- 구조가 단순하고 저렴.

**② 옵토커플러 방식 (대부분 LOW-active)**

옵토커플러(Optocoupler)는 **LED + 포토트랜지스터**가 하나의 패키지 안에 들어 있는 부품이다. 입력 쪽 LED가 빛을 내면, 출력 쪽 포토트랜지스터가 그 빛을 감지하여 ON된다. 전기적으로는 완전히 분리(광 절연, Galvanic Isolation)되어 있고, 오직 빛으로만 신호를 전달한다.

```
옵토커플러 내부 구조 (예: PC817)

┌────────────────────────────┐
│  입력 쪽        출력 쪽     │
│                            │
│  1(A) ──▶│──  4(C)        │
│        LED  ~~~▶  포토TR   │  ~~~ = 빛 (광 결합)
│  2(K) ──▶│──  3(E)        │
│                            │
└────────────────────────────┘
핀: 1=Anode, 2=Cathode, 3=Emitter, 4=Collector
```

릴레이 모듈에서의 연결:

```
[입력 쪽 — MCU와 연결]
VCC(3.3V) ──저항──▶ 1(A) LED 2(K) ◀── GPIO

[출력 쪽 — 릴레이 코일 구동]
VCC(5V) ── 릴레이 코일 ── 4(C) 포토TR 3(E) ── GND
```

- GPIO Low → LED 양단에 전위차 발생 (VCC → 저항 → LED → GPIO=Low) → LED 발광 → 포토트랜지스터 ON → 코일에 전류 → 릴레이 동작
- GPIO High → LED 양단 전위차 없음 (양쪽 다 High) → LED 꺼짐 → 포토트랜지스터 OFF → 릴레이 해제

**왜 LOW-active가 되는가?**
LED 애노드(1번)가 VCC에 연결되어 있으므로, 캐소드(2번) 쪽을 Low로 끌어내려야 전류가 흐른다. 따라서 GPIO Low = LED ON = 릴레이 ON이 된다.

**광 절연의 장점:**
- MCU 쪽 회로와 릴레이/부하 쪽 회로가 전기적으로 완전히 분리됨
- 릴레이 코일이 꺼질 때 발생하는 역기전력(Back-EMF)이나 고전압 부하의 노이즈가 MCU로 전달되지 않음
- 산업용, 고전압(AC 220V 등), 고전류 환경에서 MCU 보호에 필수적

**정리:**

| 방식 | Active 방향 | 특징 |
|------|------------|------|
| NPN 트랜지스터 | HIGH-active | 단순, 저렴, MCU-릴레이 전기적 연결됨 |
| 옵토커플러 | 대부분 LOW-active | 광 절연으로 MCU 보호, 고전압 환경에 적합 |

> 모듈 PCB를 보면 구분할 수 있다: 검은색 4핀 칩(옵토커플러, 예: PC817)이 보이면 옵토커플러 방식, 3핀 트랜지스터(예: S8050)만 보이면 트랜지스터 방식이다.

### 3. 핸들 패턴 — C에서 다중 인스턴스 관리

C는 클래스/객체가 없으므로, 동일한 하드웨어가 여러 개일 때 **구조체 + 함수 첫 인자 포인터** 패턴으로 인스턴스를 구분한다.

| 개념 | C++ (OOP) | C (핸들 패턴) |
|------|-----------|--------------|
| 인스턴스 | `Relay relay1;` (객체) | `Relay_Handle relay1;` (구조체 변수) |
| 멤버 변수 | `this->port` | `relay->port` |
| 메서드 호출 | `relay1.set(ON)` | `Relay_Set(&relay1, ON)` |

STM32 HAL 전체가 이 패턴을 사용한다: `HAL_ADC_Start(&hadc1)`, `HAL_I2C_Init(&hi2c1)` 등.

### 4. BSP 헤더에서 `#include "main.h"` vs 직접 HAL 헤더 include

- `main.h`를 include하면 편리하지만, 해당 BSP가 **특정 프로젝트의 CubeMX 설정에 의존**하게 된다.
- 핸들 패턴처럼 핀 매크로를 사용하지 않는 드라이버는 필요한 HAL 헤더만 직접 include하면 **다른 프로젝트에서도 재사용** 가능하다.

### 5. SPI1_NSS 등 핀 겸용 기능(Alternate Function) 확인

STM32 핀은 대부분 여러 기능을 겸용한다. GPIO로 사용하려는 핀이 SPI, I2C, UART 등의 Alternate Function과 겹치는지 확인해야 한다. PA4는 SPI1_NSS 겸용이므로, 나중에 SPI 장치를 추가할 가능성을 고려하면 피하는 것이 안전하다. 데이터시트의 Alternate Function Mapping 테이블에서 확인 가능하다.
