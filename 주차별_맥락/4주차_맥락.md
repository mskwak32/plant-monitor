- 15년 이상 된 센서라 노화(±1%RH/year) 반영하여 5회 중간값 필터 적용
- 온습도는 임계값 제어에 사용하지 않으므로 오차는 허용 범위

## 주요 구현 내용

### delay_us (DWT 기반)

```c
static void delay_us(uint32_t us) {
    uint32_t start = DWT->CYCCNT;
    uint32_t ticks = us * (SystemCoreClock / 1000000U);
    while ((DWT->CYCCNT - start) < ticks);
}
```

- HAL_Delay()는 ms 단위라 수십 us 타이밍 제어 불가 → DWT CYCCNT 사용
- 뺄셈 방식으로 32비트 overflow에도 올바르게 동작
- DWT 활성화는 main.c USER CODE BEGIN 2에서 수행

### DWT 활성화 (main.c USER CODE BEGIN 2)

```c
CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
DWT->CYCCNT = 0;
DWT->CTRL   |= DWT_CTRL_CYCCNTENA_Msk;
```

### 통신 프로토콜 요약

```
MCU → 센서: 5ms LOW → 30us HIGH (Start Signal)
센서 → MCU: 80us LOW → 80us HIGH (응답)
데이터:     50us LOW → 26~28us HIGH = "0" / 70us HIGH = "1"
총 40비트:  16bit 습도 + 16bit 온도 + 8bit 체크섬
```

### 데이터 변환

```c
data->humidity    = bytes[0] + bytes[1] * 0.1f;
// 온도 MSB = 1이면 영하
if (bytes[2] & 0x80)
    data->temperature = -((bytes[2] & 0x7F) + bytes[3] * 0.1f);
else
    data->temperature = bytes[2] + bytes[3] * 0.1f;
```

### 5회 중간값 필터

```c
HAL_StatusTypeDef RHT01_ReadMedian(RHT01_Data *data);
```

- 5회 읽기 (2초 간격 = 총 약 10초)
- hum 기준 버블 정렬 후 중간값 선택 (temp는 hum과 쌍으로 이동)
- 1회 이상 성공하면 OK, 전부 실패 시 HAL_ERROR 반환

## 이번 주 배운 것들

### 비트뱅잉 (Bit-banging)

MCU의 GPIO 핀을 소프트웨어로 직접 HIGH/LOW 제어하여 통신 프로토콜을 구현하는 방식. RHT-01처럼 전용 하드웨어 인터페이스(I2C, SPI 등)를 쓰지 않는 센서를 연결할 때 사용한다. 핀을 Output으로 바꿔 신호를 보내고, Input으로 바꿔 센서 응답을 읽는 과정을 코드로 직접 타이밍에 맞춰 제어한다. 정밀한 타이밍이 중요하기 때문에 HAL_Delay()처럼 ms 단위 딜레이로는 부족하고, us 단위 제어가 필요하다.

### DWT (Data Watchpoint and Trace)

ARM Cortex-M 코어에 내장된 디버그/성능 측정 하드웨어 유닛. 그 중 `DWT->CYCCNT`는 CPU 클럭이 1번 뛸 때마다 1씩 증가하는 32비트 카운터다. 72MHz 클럭에서는 1us = 72 ticks이므로, 원하는 us를 ticks으로 변환하여 경과 여부를 폴링하면 us 단위 딜레이를 구현할 수 있다. `CoreDebug->DEMCR`로 DWT 유닛을 활성화하고 `DWT->CTRL`로 CYCCNT 카운팅을 시작해야 한다.

```c
// 활성화
CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
DWT->CYCCNT = 0;
DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;

// 사용
uint32_t start = DWT->CYCCNT;
uint32_t ticks = us * (SystemCoreClock / 1000000U);
while ((DWT->CYCCNT - start) < ticks);  // 뺄셈 방식 → overflow-safe
```

뺄셈 방식(`CYCCNT - start`)이 중요한데, CYCCNT가 32비트 최댓값에서 0으로 오버플로우되더라도 부호 없는 정수 뺄셈에서 자동으로 wrap-around되어 올바른 경과 시간을 계산한다.

### HAL_ADC_PollForConversion

ADC 변환이 완료될 때까지 CPU가 그 자리에서 기다리는 함수. "폴링" 방식으로, CPU가 계속 "변환 끝났어?" 하고 반복 확인한다. `HAL_ADC_Start()`로 변환을 시작한 뒤 이 함수를 호출하지 않으면, 변환이 끝나기 전에 `HAL_ADC_GetValue()`를 호출하게 되어 이전 변환의 오래된 값을 읽는 문제가 생긴다. 두 번째 인자는 타임아웃(ms)으로, `HAL_MAX_DELAY`는 사실상 무한 대기라 센서 고장 시 프로그램이 영구적으로 멈출 수 있다. 실제 제품에서는 적절한 타임아웃 값을 지정하는 것이 좋다.

### HAL_ADCEx_Calibration_Start

STM32F1 ADC는 칩 내부의 제조 공정 오차로 인해 오프셋 오차가 존재할 수 있다. 이 함수는 MCU가 부팅 직후 내부적으로 기준값을 측정하고 오차 보정값을 ADC 레지스터에 저장하는 1회성 캘리브레이션을 수행한다. `MX_ADC1_Init()` 이후, 실제 ADC 사용 전에 한 번만 호출하면 된다. 함수 이름의 `Ex`는 Extended를 뜻하며, STM32F1 계열 전용이다(F4/F7 등은 캘리브레이션 방식이 다름). 생략해도 대부분 동작하지만 측정값에 미세한 오프셋 오차가 남을 수 있다.

### GPIO_MODE_OUTPUT_PP (Push-Pull)

GPIO 출력 모드의 한 종류. Push-Pull은 핀이 HIGH를 출력할 때 VDD(3.3V)로 핀을 밀어올리고(Push), LOW를 출력할 때 GND로 끌어내리는(Pull) 방식이다. 핀이 능동적으로 양방향 전류를 공급하므로 출력이 확실하고 응답 속도가 빠르다.

반대 방식인 **Open-Drain**은 LOW만 능동적으로 내리고, HIGH 상태는 핀을 고임피던스로 두어 외부 풀업 저항에 의존한다. 여러 장치가 같은 신호선을 공유하는 I2C 버스에서 주로 사용된다. RHT-01의 데이터 핀은 원칙적으로 Open-Drain 방식이지만, 우리 구현에서 Start Signal을 전송할 때는 MCU→센서 단방향이므로 Push-Pull로도 정상 동작한다.

이 설정을 빠뜨리면 GPIO 기본값인 Input 모드가 유지되어 `HAL_GPIO_WritePin()`을 호출해도 핀에 아무 변화가 없다.

### 레이어드 아키텍처 (Layered Architecture)

펌웨어를 역할에 따라 계층으로 분리하는 설계 방식. 프로젝트가 커질수록 `main.c` 한 파일에 센서 드라이버, 통신 코드, 비즈니스 로직이 뒤섞이면 유지보수가 어려워지는 문제를 해결한다.

```
App  →  BSP  →  HAL(Drivers)  →  하드웨어
```

상위 레이어는 하위 레이어를 호출할 수 있지만 역방향은 허용하지 않는다는 것이 핵심 원칙이다.

| 레이어 | 역할 | 예시 |
|--------|------|------|
| Core/ | CubeMX 자동생성 영역. 건드리지 않음 | main.c(초기화), stm32f1xx_it.c |
| BSP/ (Board Support Package) | 특정 하드웨어에 종속된 드라이버 | soil_sensor.c, rht01.c |
| App/ | 하드웨어를 몰라도 되는 비즈니스 로직 | plant_monitor.c (향후) |
| Drivers/ | CubeMX 자동생성 HAL / CMSIS | stm32f1xx_hal_adc.c 등 |

BSP 레이어를 분리하면 나중에 보드가 교체되거나 센서 핀이 바뀌어도 BSP 파일만 수정하면 되고, App 로직은 손대지 않아도 된다.

### STM32CubeIDE 빌드 경로 설정

새 폴더를 추가한 후 IDE가 소스와 헤더를 인식하려면 두 가지 설정이 필요하다.

**Include Path 추가** (헤더 파일 탐색 경로)
- Properties → C/C++ Build → Settings → MCU/MPU GCC Compiler → Include paths
- `../BSP/Inc` 추가

**Source Location 추가** (빌드할 .c 파일 탐색 경로)
- Properties → C/C++ General → Paths and Symbols → Source Location 탭
- `Add Folder...`로 `BSP` 폴더 추가
- STM32CubeIDE는 하위 폴더까지 재귀적으로 탐색하므로 `BSP/Src`가 아닌 `BSP` 전체를 추가하면 된다

Include Path만 추가하고 Source Location을 빠뜨리면, 헤더는 찾지만 .c 파일은 컴파일되지 않아 링커 에러(`undefined reference`)가 발생한다.

## App 레이어는 언제 채워지나?

현재 App/ 폴더는 비어있다. 지금은 "센서 읽어서 UART 출력"이 전부라 BSP 함수를 main.c에서 직접 호출해도 복잡하지 않다. 향후 릴레이/펌프 제어, OLED 표시, RPi와의 UART 통신 로직이 추가되면 `App/Src/plant_monitor.c` 같은 파일을 만들어 분리할 계획이다.

```c
// 미래의 App/Src/plant_monitor.c 예시
void PlantMonitor_Run(void) {
    SoilSensor_Data soil;
    RHT01_Data rht;

    SoilSensor_Read(&soil);
    RHT01_ReadMedian(&rht);

    if (soil.adc_value > 3000) {
        Relay_On();   // BSP 함수 호출
    }
    OLED_Display(&soil, &rht);  // BSP 함수 호출
}
```

`PlantMonitor_Run()`은 "수분이 낮으면 물을 줘라"는 로직만 알고, ADC 레지스터나 GPIO 핀 번호는 전혀 몰라도 된다.
