## 버그: STM32 재연결 시 설정값이 OLED에 반영 안 됨

### 증상

| 케이스 | 동작 |
|---|---|
| 서버 재시작 (STM32 연결 유지) | OLED 설정값 정상 반영 ✅ |
| 웹 대시보드 설정값 변경 | OLED 정상 반영 ✅ |
| STM32 USB 재연결 | OLED 기본값 유지, 서버 설정값 미반영 ❌ |

### 원인

기존 방식은 STM32 연결 후 고정 sleep(5초) 뒤 threshold를 전송했다. USB 재연결 케이스에서는:

1. STM32 cold boot → `PlantMonitor_Init()` 완료 전에 포트가 열림
2. RPi가 1초(또는 5초) 후 threshold 전송
3. 이 시점에 `UartCmd_Init()` → `HAL_UART_Receive_IT()`가 아직 활성화되지 않았을 수 있어 threshold가 유실됨

서버 재시작 케이스가 동작하는 이유: STM32는 이미 running 상태이므로 UART 인터럽트가 활성화된 채로 threshold를 수신할 수 있다.

### 해결: ping/ready handshake

고정 sleep 대신 STM32가 준비됐다는 신호를 받은 후 동기화하는 구조로 변경.

**STM32 변경 (`plant_monitor.c`)**:
- `PlantMonitor_Init()` 완료 직후 `msg={"type":"ready"}` 전송
- `PlantMonitor_HandleUartCmd()`에서 ping 수신 시 `msg={"type":"ready"}` 응답

**RPi 변경 (`protocol.py`)**:
- `ReadySignal` 타입 추가
- `get_ping_message()` 함수 추가
- `parse_line()`에서 `type=ready` 메시지 → `ReadySignal` 반환

**RPi 변경 (`uart_listener.py`)**:
- 연결 후 1초 대기 → ping 전송
- read loop에서 `ReadySignal` 수신 시 `_on_connected()` 호출 → threshold 동기화
- `ReadySignal`은 SSE 큐에 넣지 않고 `continue`

### 케이스별 동작

| 케이스 | 흐름 |
|---|---|
| 서버 재시작 | STM32 running → ping 수신 → ready 응답 → threshold 동기화 |
| STM32 리셋 버튼 | 포트 유지 → Init ready 전송 → read loop에서 수신 → 동기화 |
| USB 재연결 | Init ready 유실 가능 → ping 재전송으로 보완 |

## 버그: USB 재연결 시 ping 전송 후 ready 미수신

### 증상

USB 재연결 후 "ping 전송" 로그에서 멈춤. "STM32 ready 수신" 미출력.

### 원인

USB cold boot 시 `/dev/ttyACM0`이 등장하는 시점과 STM32 UART 초기화 완료 시점이 다르다. 포트가 열릴 때 `PlantMonitor_Init()`의 ready가 이미 전송됐거나(포트 열기 전 유실), ping이 UART 초기화 전에 도달해서 수신이 안 될 수 있다.

### 해결: ping 재전송

ready 수신 전까지 3초마다 ping을 재전송하는 로직을 read loop에 추가.

```python
_synced = False
_last_ping = time.time()

while True:
    line = port.readline()   # timeout=1s
    if not line:
        if not _synced and time.time() - _last_ping >= _PING_RETRY_INTERVAL:
            port.write(get_ping_message())
            logger.info("ping 재전송")
            _last_ping = time.time()
        continue

    ...
    if isinstance(parsed, ReadySignal):
        _synced = True
        _on_connected()   # threshold 동기화
        continue
```

`readline()`의 1초 타임아웃을 자연스러운 tick으로 활용해 별도 스레드 없이 구현.

## 이번 주 배운 것들

### 1. 고정 sleep 대기의 한계와 handshake 패턴

임베디드 통신에서 "N초 기다렸다가 전송"은 타이밍이 불확실할 때 근본적으로 불안정하다. 상대방이 준비됐다는 신호를 직접 받은 후 동작하는 **handshake 패턴**이 올바른 접근이다.

이 프로젝트의 핵심:
- STM32: 준비 완료 시 ready 전송 + ping 응답
- RPi: ready 수신 시 동기화 실행

handshake의 두 가지 트리거를 모두 구현해야 세 케이스(서버 재시작 / 리셋 버튼 / USB 재연결)를 커버할 수 있다.

---

### 2. 재전송(retry) 설계 원칙

신호가 유실될 수 있는 환경에서는 단발성 전송이 아닌 **ACK를 받을 때까지 재전송**하는 구조가 필요하다.

핵심 요소:
- **재전송 조건**: `not _synced` — ACK(ready) 미수신 상태일 때만 재전송
- **재전송 간격**: 너무 짧으면 중복 처리 위험, 너무 길면 복구가 늦음 → 3초
- **중단 조건**: `_synced = True` — ACK 수신 즉시 재전송 중단

`_synced` 플래그는 연결이 끊길 때마다 `False`로 리셋되어야 다음 재연결 시에도 동작한다 (outer while 재진입 시 `_synced = False`로 초기화됨).

---

### 3. 스레드에서 asyncio Queue에 데이터 넣는 방법

`uart_listener.py`의 read loop은 **일반 스레드**에서 실행되고, SSE 엔드포인트는 **asyncio 이벤트 루프**에서 실행된다. 두 실행 컨텍스트 간 데이터 전달에 `loop.call_soon_threadsafe(queue.put_nowait, parsed)`를 사용한다.

- `queue.put_nowait(parsed)` 만으로는 스레드 안전하지 않음 (asyncio Queue는 스레드 세이프하지 않음)
- `call_soon_threadsafe`는 이벤트 루프에 콜백을 예약하는 스레드 안전 방법

`ReadySignal`은 웹 클라이언트에 전달할 데이터가 아니므로 큐에 넣지 않고 `continue`로 건너뛴다. `stream.py`에 `else: continue`가 있어 실수로 들어가도 크래시는 나지 않지만, 의도적으로 배제하는 게 명확한 설계다.

### 4. systemd로 FastAPI 서버 자동 실행 등록

RPi 부팅 시 FastAPI 서버가 자동 시작되도록 systemd 서비스로 등록한다.

핵심 설정:
```ini
[Unit]
After=network.target   # 네트워크 준비 후 시작

[Service]
WorkingDirectory=...
ExecStart=.venv/bin/uvicorn main:app ...
Restart=on-failure     # 비정상 종료 시 자동 재시작

[Install]
WantedBy=multi-user.target
```

주요 명령:
```bash
sudo systemctl enable plant-monitor   # 부팅 시 자동 시작 등록
sudo systemctl start plant-monitor    # 즉시 시작
sudo systemctl status plant-monitor   # 상태 확인
journalctl -u plant-monitor -f        # 로그 실시간 확인
```
