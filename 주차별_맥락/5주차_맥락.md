## HAL I2C 주소 표기

```
7비트 주소 (데이터시트):  0 1 1 1 1 0 0      = 0x3C
8비트 주소 (HAL용):      0 1 1 1 1 0 0 0  = 0x78  (← << 1 한 값)
                                       └─ R/W 비트 (HAL이 내부에서 0/1 세팅)
```

모듈 뒷면에 0x78로 표기 → 이미 8비트 형식 → HAL에 0x78 그대로 사용

```c
// 추가 << 1 불필요
HAL_I2C_Master_Transmit(&hi2c1, 0x78, pData, size, HAL_MAX_DELAY);
```

### I2C (Inter-Integrated Circuit) 통신

1982년 Philips(현 NXP)가 개발한 직렬 통신 프로토콜. 선 2개(SCL, SDA)만으로 여러 장치를 하나의 버스에 연결할 수 있다.

**UART와의 차이:**

```
UART 방식 — 장치마다 선 2개 필요:
MCU ──TX/RX──▶ OLED
MCU ──TX/RX──▶ 온도센서
MCU ──TX/RX──▶ EEPROM

I2C 방식 — 선 2개로 전부:
MCU ──SDA/SCL──┬──▶ OLED     (주소 0x3C)
               ├──▶ 온도센서  (주소 0x48)
               └──▶ EEPROM   (주소 0x50)
```

#### 핀 2개의 역할

| 핀 | 이름 | 역할 |
|----|------|------|
| SCL | Serial Clock | Master가 생성하는 클럭. 모든 장치가 이 박자에 맞춰 통신 |
| SDA | Serial Data | 데이터를 주고받는 양방향 신호선 |

#### Open-Drain + 풀업 저항

I2C 선은 **Open-Drain** 방식이다. 각 장치는 선을 LOW로 끌어내리는 것만 할 수 있고, HIGH는 풀업 저항이 VCC로 올린다.

```
VCC (3.3V 또는 5V)
 │
[풀업 R]  ← 보통 4.7kΩ
 │
SDA ────┬────┬────┬────
        │    │    │
      [MCU] [장치A] [장치B]
        각 장치는 LOW로만 당길 수 있음
```

**왜 Open-Drain인가?**
여러 장치가 같은 선을 공유하기 때문이다. 한 장치가 HIGH를 출력하고 다른 장치가 LOW를 출력하면 단락(short)이 생긴다. Open-Drain은 "LOW로 당기는 것"만 하므로 충돌이 없다. 아무도 안 당기면 풀업이 자연스럽게 HIGH로 올린다.

#### 7비트 주소 체계

I2C 버스에 연결된 장치는 7비트 고유 주소로 구분한다. Master가 "0x3C야 응답해"라고 하면 해당 주소의 장치만 응답한다.

```
이론상 한 버스에 127개 장치 연결 가능 (0x00 예약 제외)

이 프로젝트 예시:
SSD1306 OLED → 0x3C
(추후 추가 장치가 있다면 주소 충돌 없는 것 사용)
```

주소는 하드웨어 핀(SA0 등)으로 결정된다. 모듈 뒷면 저항 납땜 위치로 변경 가능. 같은 주소의 장치가 2개 이상이면 통신 불가.

---

#### I2C 통신 과정 (한 번의 쓰기)

```
[S][주소 7bit + W=0][A][데이터 8bit][A][데이터 8bit][A][P]
 │                  │               │             │   │
START           Slave ACK       Slave ACK   Slave ACK STOP
```

**단계별:**

**① START 조건**

SCL이 HIGH인 상태에서 SDA를 HIGH → LOW로 내린다.

```
SCL  ‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾\_____
SDA  ‾‾‾‾‾\___________________
           ↑
         START
```

"지금부터 버스를 사용하겠다"는 선언. 다른 Master가 이미 버스를 쓰고 있으면 SCL이 LOW이므로 START 조건이 만들어지지 않는다 (자동 중재).

**② 주소 + R/W 비트 전송 (9비트)**

7비트 주소 + 1비트(R/W: 0=쓰기, 1=읽기)를 전송. SDA에 비트를 올리고 SCL을 HIGH로 올리면 Slave가 SCL 상승 엣지에서 SDA를 읽는다.

```
SCL  __/‾\_/‾\_/‾\_/‾\_/‾\_/‾\_/‾\_/‾\_____
SDA  --[0][1][1][1][1][0][0][W][ ACK ]------
         7  6  5  4  3  2  1  0
         ← 주소 0x3C (0111100) → → R/W →
```

SCL이 HIGH인 구간에서 SDA를 읽고, SCL이 LOW인 구간에서 SDA를 바꾼다.

**③ ACK (Acknowledge)**

8번째 비트 전송이 끝나고 SCL이 LOW일 때 Slave가 SDA를 LOW로 당긴다. 
그 후 Master가 SCL을 HIGH로 올리고 SDA가 LOW인지 확인한다. = ACK.
아무도 안 당기면 SDA가 풀업으로 HIGH = NACK (해당 주소 장치 없음 또는 오류).

```
SCL  ____/‾‾‾\_____
SDA  ‾‾‾‾[LOW]‾‾‾    ← Slave가 SDA를 LOW로 당김 = ACK
          ↑
          ACK 구간
```

**④ 데이터 전송 (9비트 단위 반복)**

8비트 데이터 → ACK 순서 반복.
- 쓰기 (W=0): Master → Slave, Slave가 ACK
- 읽기 (R=1): Slave → Master, Master가 ACK (마지막은 NACK)

**⑤ STOP 조건**

SCL이 HIGH인 상태를 유지하면서 SDA를 LOW → HIGH로 올린다.

```
SCL  _______/‾‾‾‾‾‾‾‾‾‾‾
SDA  ___________/‾‾‾‾‾‾‾
                ↑
              STOP
```

이 패턴은 일반 데이터 전송 중(SCL HIGH 동안 SDA 안정)에는 절대 나오지 않으므로, 버스 해제 신호로 인식된다.

---

#### SCL과 SDA의 관계 — 한 비트의 타이밍

```
      ┌── 데이터 준비 ──┐  ┌── 읽기 ──┐  ┌── 다음 준비 ──┐
SCL   ________/‾‾‾‾‾‾‾‾‾\________/‾‾‾‾‾‾‾‾‾\________
SDA   ====[안정된 비트값]====     =====[안정]=====
                   ↑                   ↑
             상승 엣지에서         상승 엣지에서
             SDA 샘플링            SDA 샘플링
```

- **SCL LOW 구간**: 송신측이 SDA를 바꿔도 되는 시간 (데이터 준비)
- **SCL 상승 엣지 (↑)**: 수신측이 SDA를 읽는 순간
- **SCL HIGH 구간**: SDA가 안정적으로 유지되어야 함

START/STOP은 예외 — SCL HIGH 중에 SDA가 변하는 특수 조건이다.

#### I2C 속도 모드

| 모드 | 속도 | 설명 |
|------|------|------|
| Standard Mode | 100 kHz | 기본값, 대부분 센서 지원 |
| Fast Mode | 400 kHz | SSD1306 지원, 화면 갱신 빠름 |
| Fast Mode Plus | 1 MHz | 고속 장치용 |
| High Speed | 3.4 MHz | 특수 목적 |

SSD1306은 Fast Mode(400kHz)까지 지원한다 (tcycle 최솟값 2.5μs → 1/2.5μs = 400kHz).
CubeMX 기본값 100kHz로도 충분하지만, OLED 갱신 속도가 느리면 400kHz로 변경 가능.

#### STM32 HAL I2C 함수

```c
// 쓰기: DevAddress 장치에 pData 전송
HAL_I2C_Master_Transmit(&hi2c1, DevAddress, pData, Size, Timeout);

// 읽기: DevAddress 장치에서 pData 수신
HAL_I2C_Master_Receive(&hi2c1, DevAddress, pData, Size, Timeout);
```

DevAddress는 **이미 1비트 왼쪽으로 시프트된 8비트 주소**를 받는다.

```c
// 데이터시트 표기가 7비트 0x3C인 경우:
HAL_I2C_Master_Transmit(&hi2c1, 0x3C << 1, ...);   // = 0x78

// 데이터시트/모듈 표기가 8비트 0x78인 경우:
HAL_I2C_Master_Transmit(&hi2c1, 0x78, ...);         // 그대로 사용
```

### Alternate Function (AF) 모드

STM32 GPIO 핀 하나가 여러 역할을 할 수 있도록 설계되어 있다.

```
PB6 핀이 할 수 있는 일:
 ├── 일반 GPIO 출력  → HAL_GPIO_WritePin()으로 코드에서 직접 제어
 ├── 일반 GPIO 입력  → HAL_GPIO_ReadPin()으로 읽기
 └── AF: I2C1_SCL   → I2C 하드웨어가 핀을 직접 제어
```

**일반 GPIO 모드:** 코드에서 핀을 직접 HIGH/LOW 제어. RHT-01 비트뱅잉이 이 방식.

**AF 모드:** 핀의 제어권을 내부 하드웨어 블록(I2C, UART, SPI 등)에 넘긴다. `HAL_I2C_Master_Transmit()`을 호출하면 I2C 하드웨어가 알아서 SCL/SDA 파형을 만들어낸다. 코드에서 핀을 직접 건드리지 않는다.

```
일반 GPIO:  코드 ──▶ GPIO 레지스터 ──▶ 핀 출력
AF 모드:    코드 ──▶ I2C 하드웨어 ──▶ 핀 출력
                    (타이밍 자동 생성)
```

**AF_OD vs AF_PP:**

I2C는 Open-Drain 방식이 필요하므로 `GPIO_MODE_AF_OD` 사용.
UART는 1:1 통신이라 충돌 걱정이 없으므로 `GPIO_MODE_AF_PP` 사용.

```c
// stm32f1xx_hal_msp.c에서 확인 가능

// I2C1 (PB6, PB7)
GPIO_InitStruct.Mode = GPIO_MODE_AF_OD;   // Alternate Function Open-Drain

// USART2 (PA2, PA3)
GPIO_InitStruct.Mode = GPIO_MODE_AF_PP;   // Alternate Function Push-Pull
```

**소스코드에서 핀 매핑 확인하는 곳:** `stm32f1xx_hal_msp.c`의 `HAL_I2C_MspInit()` 함수. CubeMX가 주석으로 명시해준다:

```c
/**I2C1 GPIO Configuration
PB6     ------> I2C1_SCL
PB7     ------> I2C1_SDA
*/
```

### 5V Tolerant (FT) 핀

**확인 방법:** STM32F103RB 데이터시트 → **Table: Pin definitions** → **I/O Level** 컬럼

| 표시 | 의미 |
|------|------|
| FT | Five-volt Tolerant: 5V 입력 허용 |
| 표시 없음 | 3.3V 초과 입력 시 손상 위험 |

PB6, PB7 모두 **FT** 핀이다.

**중요:** FT는 **입력 내성**에 대한 것이다. 출력은 항상 3.3V.

```
FT 핀에 5V 신호가 들어오는 것  → ✅ 안전
FT 핀이 5V를 출력하는 것       → ❌ 불가 (출력은 항상 3.3V)
```

I2C는 Open-Drain이므로 HIGH 출력을 풀업 저항이 담당한다. MCU가 "직접 5V를 출력"하는 것이 아니라 선을 놓아주면 풀업이 당기는 방식이므로, FT 핀이 5V 풀업 버스에 연결되는 것이 가능하다.

### I2C 주소 선택 (SA0)

SSD1306은 D/C# 핀(= SA0 핀)의 레벨로 I2C 주소를 결정한다:
- D/C# = GND → 7비트 주소 `0111100b` = **0x3C** (HAL: 0x78)
- D/C# = VDD → 7비트 주소 `0111101b` = **0x3D** (HAL: 0x7A)

모듈 뒷면의 저항 납땜 위치가 이 SA0 핀의 연결을 결정한다. 같은 모듈 2개를 한 버스에 쓰려면 하나를 0x7A로 리워크해야 한다.

### SSD1306이란

SSD1306은 **OLED 패널을 구동하는 전용 IC(드라이버 칩)**다. OLED 픽셀은 하나하나에 정확한 전압/전류를 인가해야 빛나는데, MCU가 직접 수백 개 픽셀을 아날로그로 제어하기는 핀 수도 부족하고 타이밍도 복잡하다.

```
STM32 ──I2C (2선)──▶ SSD1306 ──수백 개 내부 선──▶ 128×64 OLED 픽셀
  (디지털 명령)       (드라이버 IC)               (OLED 패널)
```

SSD1306이 하는 일:
- MCU로부터 I2C로 "어떤 픽셀을 켜라" 데이터를 받음
- 내부 Charge Pump로 OLED 구동에 필요한 고전압(7~9V) 자체 생성
- 각 픽셀에 적절한 전류 공급

MCU가 하는 일은 "데이터 전달"뿐이고, 실제 픽셀 구동은 SSD1306이 전담한다.

### GDDRAM — SSD1306 내부 화면 메모리

SSD1306 내부에는 **GDDRAM(Graphic Display Data RAM)** 이라는 1024바이트짜리 픽셀 전용 메모리가 있다.

```
GDDRAM = 128 × 64 비트 = 1024 바이트

비트 1개 = 픽셀 1개
  1 → 해당 픽셀 ON (빛남)
  0 → 해당 픽셀 OFF (꺼짐)
```

SSD1306은 이 메모리를 주기적으로 읽어서 화면에 뿌린다. **"화면에 그린다" = "GDDRAM에 쓴다"** 가 핵심이다.

레지스터와의 비교:
- 레지스터는 특정 주변장치의 **동작 방식을 설정**하는 4~8바이트짜리 메모리
- GDDRAM은 **현재 화면 상태 전체를 저장**하는 1024바이트짜리 메모리
- 공통점: "특정 주소에 값을 쓰면 하드웨어가 반응한다"

#### GDDRAM 구조 — Page 방식

SSD1306은 행(row) 8개를 묶어서 **Page**라고 부른다. 8비트(1바이트)로 처리하기 편하기 때문이다.

```
Page 0 : Row 0~7   (y=0~7)
Page 1 : Row 8~15  (y=8~15)
Page 2 : Row 16~23 (y=16~23)
Page 3 : Row 24~31 (y=24~31)
Page 4 : Row 32~39 (y=32~39)
Page 5 : Row 40~47 (y=40~47)
Page 6 : Row 48~55 (y=48~55)
Page 7 : Row 56~63 (y=56~63)
```

각 Page는 128바이트. 한 바이트 = 세로 8픽셀:

```
바이트의 bit 0 (LSB) → 위쪽 픽셀 (row 0)
바이트의 bit 7 (MSB) → 아래쪽 픽셀 (row 7)

예: 0b00000101
           │ │
           │ └─ row 0 켜짐
           └── row 2 켜짐
      76543210  <- 비트번호 (오른쪽이 bit0, LSB)     
```

#### 픽셀 (x, y)를 framebuffer에서 찾는 공식

```c
page   = y / 8;      // 어느 Page인가
bit    = y % 8;      // Page 안에서 몇 번째 비트인가
column = x;          // 어느 column인가

// 켜기
buffer[page * 128 + column] |= (1 << bit);

// 끄기
buffer[page * 128 + column] &= ~(1 << bit);
```

`|=` 로 켜고 `&= ~` 로 끄는 이유: `=` 로 덮어쓰면 같은 바이트의 다른 픽셀(다른 행)이 지워진다.

### SSD1306 I2C 통신 구조

SSD1306에 I2C로 보내는 데이터는 두 종류:

| 종류 | 설명 |
|------|------|
| **Command** | 초기화, 모드 설정, 커서 이동 등 |
| **Data** | GDDRAM에 쓰는 픽셀 데이터 |

전송 형식:
```
[START][주소 0x78][W][ACK] [Control byte][ACK] [데이터][ACK] ... [STOP]
```

**Control byte** — 이후 바이트들의 용도를 선언하는 헤더:

| Control byte | Co(bit7) | D/C#(bit6) | 의미 |
|---|---|---|---|
| `0x00` | 0 | 0 | 이후 모든 바이트 = Command 연속 스트림 |
| `0x40` | 0 | 1 | 이후 모든 바이트 = Data 연속 스트림 |
| `0x80` | 1 | 0 | Command 1바이트, 다음 또 Control byte |
| `0xC0` | 1 | 1 | Data 1바이트, 다음 또 Control byte |

`Co=0`이면 한 트랜잭션(START~STOP) 안에 Control byte 하나로 뒤의 모든 바이트를 같은 용도로 처리할 수 있다. START마다 Control byte를 매번 새로 붙일 필요 없다.

```c
// 커맨드 여러 개를 한 트랜잭션으로:
uint8_t cmds[] = {0x00, 0xAE, 0xD5, 0x80, ...};  // 0x00 하나로 전부 Command
HAL_I2C_Master_Transmit(&hi2c1, 0x78, cmds, sizeof(cmds), HAL_MAX_DELAY);

// 프레임버퍼 전체를 한 트랜잭션으로:
uint8_t buf[1025];
buf[0] = 0x40;                      // 0x40 하나로 뒤의 1024바이트 전부 Data
memcpy(&buf[1], framebuffer, 1024);
HAL_I2C_Master_Transmit(&hi2c1, 0x78, buf, 1025, HAL_MAX_DELAY);
```

### SSD1306 초기화 커맨드 상세

```c
0x00,        // Control byte: 이후 모두 Command

0xAE,        // Display OFF
             // 초기화 중 화면을 끈다. 켜진 상태에서 설정 변경 시
             // 화면이 깜빡이거나 잘못된 상태로 잠깐 표시될 수 있음

0xD5, 0x80,  // Set Display Clock Divide Ratio / Oscillator Frequency
             // 화면 갱신 속도(프레임레이트) 결정. 기본값 0x80으로 충분

0xA8, 0x3F,  // Set Multiplex Ratio
             // 실제로 구동하는 행의 수 - 1. 64행 패널 → 63 = 0x3F
             // 32행 패널이면 0x1F

0xD3, 0x00,  // Set Display Offset
             // 화면 시작 위치를 세로로 몇 행 밀지. 0x00 = 안 밀음

0x40,        // Set Display Start Line = 0
             // GDDRAM의 어느 행부터 화면 맨 위로 표시할지 (스크롤 기능에 활용)

0x8D, 0x14,  // Charge Pump: Enable
             // OLED 픽셀 구동에 필요한 고전압(7~9V)을 내부에서 승압
             // 없으면 화면이 절대 안 켜짐. 0x10이면 비활성화

0x20, 0x00,  // Set Memory Addressing Mode: Horizontal
             // 열 끝 도달 시 다음 Page 처음으로 자동 이동
             // 이 모드여야 Update()에서 1024바이트를 한 번에 쭉 쓸 수 있음

0xA1,        // Set Segment Re-map
             // 0xA0 = col0→SEG0 (기본), 0xA1 = col127→SEG0 (좌우 반전 보정)
             // 모듈마다 패널이 PCB에 붙은 방향이 달라 화면이 뒤집힐 수 있음

0xC8,        // Set COM Output Scan Direction
             // 0xC0 = 위→아래 (기본), 0xC8 = 아래→위 (상하 반전 보정)

0xDA, 0x12,  // Set COM Pins Hardware Configuration
             // 패널 내부 배선 방식. 128×64 패널용 = 0x12
             // 잘못 설정하면 화면이 반만 나오거나 줄무늬 생김

0x81, 0xCF,  // Set Contrast
             // 0x00~0xFF, 클수록 밝음. 배터리 제품이면 낮게 설정

0xD9, 0xF1,  // Set Pre-charge Period
             // 픽셀 켜기 전 충전 시간. Charge Pump 사용 시 권장값

0xDB, 0x30,  // Set VCOMH Deselect Level
             // 꺼진 픽셀에 인가하는 전압 기준. 높을수록 대비 향상

0xA4,        // Entire Display ON (Resume)
             // GDDRAM 내용대로 표시. 0xA5이면 전체 켜기(테스트용)

0xA6,        // Set Normal Display
             // 0xA6 = 정상(1=켜짐), 0xA7 = 반전(1=꺼짐)

0xAF,        // Display ON
```

### SSD1306 드라이버 설계 — 프레임버퍼 패턴

MCU SRAM에 **프레임버퍼**를 두고 그리기 작업을 모두 거기에 먼저 한 뒤, `Update()`로 한 번에 전송한다.

```
MCU SRAM:  framebuffer[1024]  ← 여기서 그리기 작업
                 │
           SSD1306_Update()
                 │
SSD1306 GDDRAM: 1024 바이트  ← 실제 화면
```

픽셀 하나마다 I2C 전송하면 너무 느리고, 부분 지우기/덮어쓰기도 어렵다. 프레임버퍼에 모아두고 한 번에 보내면 효율적이다.

#### SSD1306_Update 구조

```c
// ① 주소 설정: 열 0~127, Page 0~7 (화면 전체 범위 지정)
uint8_t addr_cmds[] = {
    0x00,
    0x21, 0x00, 127,   // Set Column Address
    0x22, 0x00, 7,     // Set Page Address
};
HAL_I2C_Master_Transmit(..., addr_cmds, sizeof(addr_cmds), ...);

// ② 프레임버퍼 전송
// data_buf[0] = 0x40 (Control: Data 스트림)
// data_buf[1..1024] = framebuffer 내용
static uint8_t data_buf[1025];
data_buf[0] = 0x40;
memcpy(&data_buf[1], framebuffer_, 1024);
HAL_I2C_Master_Transmit(..., data_buf, 1025, ...);
```

addr_cmds를 매번 보내는 이유: Horizontal Addressing Mode에서 1024바이트를 다 쓰고 나면 커서가 끝(Page7, Col127)에 있다. 다음 Update 전에 (0,0)으로 리셋해야 한다.

`data_buf`를 `static`으로 선언하는 이유: 1025바이트를 스택에 매번 할당하면 스택 오버플로우 위험. `static`이면 BSS 영역(전역 메모리)에 한 번만 할당된다.

**Control byte와 framebuffer를 별도로 전송하면 안 되는 이유:**
```c
// ❌ 트랜잭션이 2번 → GDDRAM 주소가 리셋됨
HAL_I2C_Master_Transmit(..., &control, 1, ...);       // 트랜잭션 1 (STOP)
HAL_I2C_Master_Transmit(..., framebuffer, 1024, ...); // 트랜잭션 2 (새 START)
```
두 번째 Transmit은 새 START를 만들고, SSD1306은 주소 커서를 다시 0으로 리셋한다. 그래서 반드시 1025바이트를 붙여서 한 번에 전송해야 한다.

### 5×8 폰트

5×8 이름의 의미:
- **5** = 가로 픽셀 수 (열, column)
- **8** = 세로 픽셀 수 (행, row)

8이 선택된 이유: SSD1306의 Page가 정확히 8행이기 때문. 폰트 한 글자 높이를 8픽셀로 맞추면 1 Page = 1 글자 행이 딱 맞아 비트 연산 없이 바이트를 그대로 복사할 수 있다.

5가 선택된 이유: 대소문자/숫자/특수문자를 구분할 수 있는 최소 가로 폭이 5픽셀. 여백 1픽셀 포함 **문자 1개 = 6픽셀**, 128 ÷ 6 = **21글자/행**.

```
'H' = {0x7F, 0x08, 0x08, 0x08, 0x7F}
       열0   열1   열2   열3   열4

각 바이트를 비트로 풀면 (bit0=행0=위, bit7=행7=아래):
  열0=0x7F  열1=0x08  열2=0x08  열3=0x08  열4=0x7F
  01111111  00001000  00001000  00001000  01111111

      열0  열1  열2  열3  열4
행0:   █    ·    ·    ·    █
행1:   █    ·    ·    ·    █
행2:   █    ·    ·    ·    █
행3:   █    █    █    █    █   ← 가로선 (0x08의 bit3)
행4:   █    ·    ·    ·    █
행5:   █    ·    ·    ·    █
행6:   █    ·    ·    ·    █
행7:   ·    ·    ·    ·    ·   ← 여백 행 (0x7F의 bit7=0)
```

폰트 테이블 인덱싱: `font5x8[ch - 0x20]`
- 테이블 인덱스 0번 = `' '`(스페이스, ASCII 0x20)
- `'A'`(0x41) → `0x41 - 0x20 = 0x21 = 33`번째 행

폰트 파일은 `ssd1306.c`에서 분리하여 별도 파일로 관리:
- `BSP/Inc/ssd1306_font.h` — `extern const uint8_t font5x8[][5];` 선언
- `BSP/Src/ssd1306_font.c` — 실제 폰트 데이터 정의

`static`을 붙이면 안 되는 이유: `ssd1306_font.c`에 `static`을 붙이면 `ssd1306.c`에서 `extern`으로 선언해도 링커가 심볼을 찾지 못한다.

### C 언어 — static 파일 전역변수와 extern

#### static의 위치에 따른 의미

```c
// 함수 안에서 → 호출 사이에 값 유지 (지역 static)
void foo(void) {
    static int count = 0;
    count++;
}

// 파일 최상단 (함수 밖) → 이 .c 파일 안에서만 접근 가능 (파일 static)
static int value = 0;
```

파일 최상단의 `static`은 "이 번역 단위(.c 파일) 전용"이라는 뜻이다. `static` 함수와 동일한 원리.

```
링커 관점:
static 없는 전역  → 심볼 테이블에 공개 (external linkage)
static 있는 전역  → 심볼 테이블에 비공개 (internal linkage)
```

#### 외부 노출이 필요한 경우 — extern

```c
// ssd1306_font.h
extern const uint8_t font5x8[][5];   // 선언만 (메모리 할당 없음)

// ssd1306_font.c
const uint8_t font5x8[][5] = { ... }; // 정의 (메모리 실제 할당, 딱 한 번)

// ssd1306.c
#include "ssd1306_font.h"
// font5x8 바로 사용 가능
```

#### 헤더에 정의를 넣으면 안 되는 이유

헤더는 `#include` 시 텍스트 그대로 복사된다. 정의를 헤더에 넣으면 include한 모든 `.c` 파일마다 메모리가 할당되어 링커 에러가 발생한다.

```
error: multiple definition of 'font5x8'
```

| 위치 | extern 여부 | 의미 |
|------|-------------|------|
| 헤더 | extern 붙임 | 선언만 — 여러 파일에 포함되어도 메모리 할당 없음 |
| .c 파일 1개 | 없음 | 정의 — 메모리 실제 할당 (딱 한 번) |
| 헤더 | extern 없음 | ❌ 정의가 여러 번 복사 → 링커 에러 |

### App 레이어 (레이어드 아키텍처 실제 적용)

4주차에서 설계만 해두었던 App 레이어를 이번 주에 실제로 채웠다.

**레이어 원칙:**

```
App  →  BSP  →  HAL(Drivers)  →  하드웨어
```

상위가 하위를 호출하는 것만 허용. 역방향은 금지.

| 레이어 | 기준 | 예시 |
|--------|------|------|
| BSP | 핀 번호, I2C 주소, ADC 채널을 알아야 함 | ssd1306.c, soil_sensor.c |
| App | 하드웨어를 몰라도 되는 로직 | plant_monitor.c |

**캡슐화 예시:**

```c
// plant_monitor.c
static uint8_t soil_threshold = 40;  // static: 이 파일 밖에서 직접 접근 불가
```

`static` 전역변수는 해당 `.c` 파일 안에서만 접근 가능하다. 외부에서 변경하려면 반드시 이 파일 안의 함수를 통해야 한다. 캡슐화를 강제하는 방법이다.

### sprintf

`String Formatted Print`의 약자. `printf`처럼 서식 지정자를 사용하되, 콘솔 대신 문자 배열(메모리)에 저장한다.

```c
char buf[24];
sprintf(buf, "Soil: %3d%%", 73);
// buf = "Soil:  73%"

SSD1306_WriteString(0, 0, buf);  // 메모리의 문자열을 OLED에 전달
```

`SSD1306_WriteString`은 메모리의 문자열을 기대하므로 `printf`(콘솔 출력)와 혼용할 수 없다. `sprintf`로 먼저 조립 후 전달한다.

**서식 지정자 정리:**

| 서식 | 의미 | 예시 |
|------|------|------|
| `%d` | 정수 | `73` |
| `%3d` | 정수, 최소 3칸 (부족하면 앞에 공백) | `" 73"` |
| `%f` | 부동소수점 | `25.300000` |
| `%5.1f` | 전체 5칸, 소수점 1자리 | `" 25.3"` |
| `%%` | 리터럴 `%` | `%` |

고정 폭(`%3d`, `%5.1f`)을 쓰는 이유: 값이 짧아질 때 이전 값의 잔상이 OLED에 남지 않도록.

### C 언어 — static 함수와 forward declaration

#### 내부 전용 함수에 static

```c
// plant_monitor.c
static void OLED_Display(uint8_t moisture, RHT01_Data *rht);
```

`static` 함수는 해당 `.c` 파일 밖에서 호출할 수 없다. 링커 심볼 테이블에 공개되지 않으므로 다른 파일에서 우연히 같은 이름 함수를 만들어도 충돌이 없다.

#### forward declaration

C는 절차지향이므로 함수를 호출하는 시점보다 위에 그 함수의 존재를 알려야 한다. 구현부를 아래에 두고 싶으면 파일 상단에 선언부만 먼저 작성한다.

```c
// 파일 상단 — 선언만 (forward declaration)
static void OLED_Display(uint8_t moisture, RHT01_Data *rht);

void PlantMonitor_Run(void) {
    OLED_Display(moisture_pct, &rht);  // ← 호출 가능
}

// 파일 하단 — 실제 구현
static void OLED_Display(uint8_t moisture, RHT01_Data *rht) {
    // ...
}
```

**선언부와 구현부 모두 `static`을 붙여야 한다.** 선언에만 붙이면 구현이 external linkage가 되어 의도와 다르게 외부에 노출된다.

#### static 함수를 헤더에 넣으면 안 되는 이유

헤더는 `#include` 시 텍스트 그대로 복사된다. 헤더에 `static void OLED_Display(...)` 선언을 넣으면 include하는 모든 `.c` 파일이 각자 복사본을 만든다. 불필요한 중복 컴파일, 메모리 낭비가 발생한다.

### C 언어 — 컴파일과 링킹 과정

`#include`가 어떻게 작동하고, 함수 구현은 어떻게 찾는가.

#### 전체 흐름

```
소스 코드 (*.c)  →  전처리  →  컴파일  →  *.o  →  링킹  →  *.elf
```

#### 1단계: 전처리 (Preprocessing)

`#include "soil_sensor.h"`를 만나면 헤더 파일 내용을 텍스트 그대로 복사해 붙여넣는다. 이 시점에서 컴파일러는 `SoilSensor_Read`의 **선언**만 알게 된다. 구현은 아직 모른다.

```c
// 전처리 후 plant_monitor.c
HAL_StatusTypeDef SoilSensor_Read(SoilSensor_Data *data);  // ← 복사됨

void PlantMonitor_Run(void) {
    SoilSensor_Read(&soil);   // "어딘가에는 있다"고만 알고 있음
}
```

#### 2단계: 컴파일 (Compilation)

각 `.c` 파일을 독립적으로 오브젝트 파일(`.o`)로 변환한다. 아직 모르는 함수 주소는 `U`(Undefined)로 표시한다.

```
plant_monitor.o:
  T PlantMonitor_Run   ← T = 이 파일에 구현 있음
  U SoilSensor_Read    ← U = 어딘가에서 찾아줘

soil_sensor.o:
  T SoilSensor_Read    ← T = 이 파일에 구현 있음
```

#### 3단계: 링킹 (Linking)

모든 `.o` 파일을 합쳐 `U`와 `T`를 매칭한다.

```
plant_monitor.o의 U SoilSensor_Read
  →  soil_sensor.o의 T SoilSensor_Read
  →  실제 주소 대입 → 최종 firmware.elf
```

구현이 없으면:
```
error: undefined reference to 'SoilSensor_Read'
```

**결론:** `#include`는 선언만 제공하고, 구현은 링커가 오브젝트 파일 전체를 뒤져서 연결한다.

---

### C 언어 — 배열 vs 구조체 포인터 전달

**배열은 자동으로 포인터로 변환된다 (array decay):**

```c
int arr[10];
func(arr);       // arr는 자동으로 &arr[0]으로 변환
```

C 언어 설계 초기, 배열 전체를 복사해서 전달하는 것은 비효율적이라 배열 이름이 자동으로 첫 번째 원소 주소로 변환되도록 특수 규칙을 만들었다.

**구조체는 자동 변환되지 않는다:**

```c
RHT01_Data rht;
func(rht);       // 구조체 전체가 복사되어 전달
func(&rht);      // 포인터(주소)가 전달됨
```

포인터를 기대하는 함수에 구조체를 그대로 넘기면 타입 불일치 오류:

```
error: incompatible type for argument 2 of 'OLED_Display'
```

**왜 구조체를 포인터로 전달하는가:**
- 복사 비용 절감 (구조체가 크면 클수록 효과적)
- 함수 내에서 원본 수정 가능 (`->` 연산자로 접근)

```c
static void OLED_Display(uint8_t moisture, RHT01_Data *rht) {
    rht->temperature   // 포인터이므로 -> 사용
}

// 호출
OLED_Display(moisture_pct, &rht);  // ← & 필수
```

### C 언어 — 변수 이름 앞/뒤 언더스코어 관행

Java에서는 멤버변수에 `_name` (앞) 또는 `mName`을 붙이는 관행이 있다. C에서는 **앞에 언더스코어를 쓰면 위험하다.**

```c
_name    // 컴파일러/표준 라이브러리 예약 식별자
__name   // 완전 예약 (절대 사용 금지)
_Name    // 대문자 시작도 예약
```

C에서 "이 파일 전용 변수"임을 표시하는 안전한 관행은 **뒤에 `_`** 를 붙이는 것이다.

```c
static I2C_HandleTypeDef *hi2c_;     // ✅ 안전
static uint8_t framebuffer_[1024];   // ✅ 안전
static I2C_HandleTypeDef *_hi2c;     // ⚠️ 피해야 함
```

### HAL_MAX_DELAY — 타임아웃 처리

`HAL_MAX_DELAY`는 `0xFFFFFFFF`(약 49일)로 정의된 사실상 무한 대기 값이다.

```c
// stm32f1xx_hal_def.h
#define HAL_MAX_DELAY  0xFFFFFFFFU
```

**학습/프로토타입용으로는 무방하나, 실제 제품에서는 쓰지 않는다.**

문제: I2C 버스에 노이즈가 끼거나 OLED가 응답 안 하면 → `HAL_I2C_Master_Transmit`이 영원히 반환 안 함 → MCU 전체가 그 줄에서 멈춤 → 워터펌프 제어 등 모든 동작 중단.

실제 제품에서는 **정상 동작 시 걸리는 시간의 2~3배**를 타임아웃으로 설정한다:

```
I2C 100kHz, 1바이트 = 9비트 × 10μs = 90μs
1025바이트 전송 ≈ 92ms → 타임아웃 200ms 설정
```

```c
#define SSD1306_TIMEOUT_MS  200

HAL_I2C_Master_Transmit(hi2c_, SSD1306_I2C_ADDR,
                        data_buf, sizeof(data_buf),
                        SSD1306_TIMEOUT_MS);
```

타임아웃 발생 시 반환값이 `HAL_TIMEOUT`이므로 에러 카운트, 재시도, 경고 표시 등의 로직을 추가한다.

현재 코드에서 `HAL_MAX_DELAY` 사용 위치 (고도화 작업으로 등록됨):

| 파일 | 함수 | 용도 |
|------|------|------|
| `BSP/Src/ssd1306.c` | `SSD1306_Init` | I2C 초기화 커맨드 전송 |
| `BSP/Src/ssd1306.c` | `SSD1306_Update` × 2 | 주소 설정 + 프레임버퍼 전송 |
| `BSP/Src/soil_sensor.c` | `SoilSensor_Read` | ADC 변환 완료 대기 |
| `Core/Src/main.c` | `__io_putchar` | UART 1바이트 전송 |

## 토양습도 센서 ADC 보정

### 실측값

| 상태 | ADC 값 |
|------|--------|
| 공기 중 (완전 건조) | ~4090 |
| 물에 완전히 담금 | ~1700 |

`soil_sensor.h`에 상수로 정의:

```c
#define SOIL_SENSOR_ADC_DRY   4090
#define SOIL_SENSOR_ADC_WET   1700
```

### 수분% 계산 공식 (선형 보간 + clamp)

```c
if (soil.adc_value >= SOIL_SENSOR_ADC_DRY) {
    moisture_pct = 0;
} else if (soil.adc_value <= SOIL_SENSOR_ADC_WET) {
    moisture_pct = 100;
} else {
    moisture_pct = (uint8_t)((SOIL_SENSOR_ADC_DRY - soil.adc_value) * 100
                             / (SOIL_SENSOR_ADC_DRY - SOIL_SENSOR_ADC_WET));
}
```

**선형 보간 공식:**

```
moisture = (DRY - adc) * 100 / (DRY - WET)

adc = 4090 → (4090 - 4090) * 100 / 2390 = 0%
adc = 2895 → (4090 - 2895) * 100 / 2390 ≈ 50%
adc = 1700 → (4090 - 1700) * 100 / 2390 = 100%
```

**clamp가 필요한 이유:**
센서를 물에 더 깊이 담그면 ADC가 1700 아래로 내려갈 수 있다. 그냥 공식을 적용하면 100%를 초과한다. `else if (adc <= WET) → 100`으로 상한을 고정한다.

**`switch`를 쓰지 않는 이유:**
`switch`는 정확한 값 하나(`case 1700:`)와만 비교할 수 있다. `>= 4090`처럼 범위 비교는 C 문법상 불가능하다. 범위 비교에는 `if-else if-else`가 올바른 선택이다.
