## 12주차 수정 내용

### 1. 펌프 로그 날짜 로컬 시간 표시

**수정 전**
```js
li.textContent = `${log.timestamp} - ${log.action}`;
```

**수정 후**
```js
const localTime = new Date(log.timestamp).toLocaleString("ko-KR");
const label = PUMP_LABEL[log.action] ?? log.action;
li.textContent = `${localTime} - ${label}`;
```

### 2. DB timestamp에 `Z` 접미사 추가

SQLite `strftime('%Y-%m-%dT%H:%M:%S', 'now')`의 `'now'`는 UTC 시각을 반환하지만 문자열에 타임존 표시가 없다. JS `new Date()`가 이를 로컬 시간으로 오해하는 문제가 발생했다.

**수정:** 포맷을 `'%Y-%m-%dT%H:%M:%SZ'`로 변경해 UTC임을 명시한다.

적용 위치:
- `database.py` — `sensor_logs`, `pump_logs`, `settings` 테이블 DEFAULT
- `repository.py` — `update_threshold()`의 `updated_at` 갱신 쿼리

> ⚠️ `CREATE TABLE IF NOT EXISTS`는 테이블이 이미 존재하면 DEFAULT 변경이 적용되지 않는다. 기존 `plant_monitor.db`를 삭제하고 서버를 재시작해야 새 스키마가 적용된다.

### 3. SSE 연결 상태 표시 제거

데이터가 화면에 표시되면 연결된 것과 동일하고, 실사용 환경에서 연결 끊김이 거의 발생하지 않는다는 판단 하에 제거했다.

제거 항목:

- `style.css` — `#connection-status`, `#connection-status.connected`, `#connection-status.disconnected` 3개 규칙
- `app.js` — `elStatus` DOM 참조, `es.onopen` 핸들러, `es.onerror` 핸들러

## 이번 주 배운 것들

---

### 1. SQLite `strftime('now')`는 UTC를 반환한다

SQLite의 `'now'` 키워드는 항상 **UTC 기준** 현재 시각을 반환한다. 이는 SQLite가 내부적으로 시간을 UTC로 관리하기 때문이다. 한국(KST, UTC+9)에서 실행해도 `'now'`는 KST가 아닌 UTC를 반환한다.

```sql
SELECT strftime('%Y-%m-%dT%H:%M:%S', 'now');
-- 결과: "2026-05-07T01:23:15" (KST 10:23:15일 때)
```

타임존 변환이 필요한 경우:
```sql
SELECT strftime('%Y-%m-%dT%H:%M:%S', 'now', 'localtime');
-- 결과: "2026-05-07T10:23:15" (시스템 로컬 시간 적용)
```

이 프로젝트에서는 DB에는 UTC(`Z` 접미사)로 저장하고, 표시는 JS에서 로컬 시간으로 변환하는 방식을 택했다. DB는 시간대에 독립적인 UTC로 유지하는 것이 표준 관행이다.

---

### 2. JS `new Date()`의 타임존 파싱 규칙

JS `new Date(문자열)`의 동작은 문자열 형식에 따라 달라진다. ISO 8601 형식 기준:

| 입력 형식 | JS 해석 |
|---|---|
| `"2026-05-07"` (날짜만) | **UTC** |
| `"2026-05-07T10:23:15"` (타임존 없음) | **로컬 시간** |
| `"2026-05-07T10:23:15Z"` (`Z` 명시) | **UTC** |
| `"2026-05-07T10:23:15+09:00"` (오프셋 명시) | **해당 오프셋** |

이 규칙 때문에 DB에서 `"2026-05-07T10:23:15"` (UTC값, `Z` 없음)를 받으면 JS가 이를 **로컬 시간으로 오해**한다. KST 환경에서는 UTC 값을 KST인 것처럼 표시하므로 9시간이 틀려진다.

**해결 방법은 두 가지다:**

**① DB 저장 시 `Z` 명시 (이 프로젝트에서 채택)**
```sql
strftime('%Y-%m-%dT%H:%M:%SZ', 'now')
-- 저장값: "2026-05-07T01:23:15Z"
```
JS가 UTC로 인식 → `.toLocaleString("ko-KR")`이 KST로 정확히 변환.

**② JS에서 `+ "Z"` 보정 (임시 대응)**
```js
new Date(log.timestamp + "Z")
```
DB 구조를 바꾸기 전 임시로 쓸 수 있지만, DB 값과 JS 코드가 암묵적으로 결합되는 문제가 있어 DB를 고치는 게 더 깔끔하다.

---

### 3. `CREATE TABLE IF NOT EXISTS`와 스키마 변경의 한계

SQLite의 `CREATE TABLE IF NOT EXISTS`는 **테이블이 이미 존재하면 아무것도 하지 않는다.** DEFAULT 값이나 컬럼 정의를 바꿔도 기존 테이블에는 적용되지 않는다.

```sql
-- 이미 테이블이 있으면 이 구문 전체가 무시됨
CREATE TABLE IF NOT EXISTS pump_logs(
    timestamp TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    ...
);
```

스키마를 변경하려면 선택지가 두 가지다:

**① DB 파일 삭제 (개발 단계에서 사용)**
```bash
rm plant_monitor.db
# 서버 재시작 시 init_db()가 새 스키마로 재생성
```

**② 마이그레이션 (운영 단계에서 사용)**
```sql
ALTER TABLE pump_logs ADD COLUMN new_col TEXT;
-- 또는 새 테이블 생성 후 데이터 이전
```

이 프로젝트는 개발 단계이므로 DB 삭제 방식을 사용했다.

---

### 4. `toLocaleString()` vs `toLocaleTimeString()` 

JS Date 객체의 로컬 포맷 변환 메서드 두 가지:

| 메서드 | 출력 예 | 용도 |
|---|---|---|
| `.toLocaleString("ko-KR")` | `"2026. 5. 7. 오전 10:23:15"` | 날짜 + 시간 모두 표시 |
| `.toLocaleTimeString("ko-KR")` | `"오전 10:23:15"` | 시간만 표시 |

펌프 로그는 날짜와 시간이 모두 필요하므로 `toLocaleString()`을 사용했다. SSE 실시간 갱신의 "마지막 갱신" 표시는 시간만 필요하므로 `toLocaleTimeString()`을 사용했다.

---

### 5. 불필요한 UI 제거 판단 기준

SSE 연결 상태 표시(`"SSE 연결됨"` / `"SSE 연결 끊김"`)를 제거한 이유:

- **데이터 표시 = 연결 확인**: 센서 수치가 화면에 갱신되고 있다면 SSE가 살아있다는 증거다. 별도 상태 텍스트가 추가 정보를 주지 않는다.
- **실사용 환경**: 같은 내부망(Wi-Fi) 안에서 동작하므로 연결 끊김이 거의 발생하지 않는다.
- **UI 단순화**: 사용자에게 의미 있는 정보만 노출하는 것이 UX에 좋다.

연결 끊김 감지가 필요한 경우라면 `es.onerror`에서 재연결 시도 로직을 넣거나, "마지막 갱신" 타임스탬프가 일정 시간 이상 갱신되지 않을 때 시각적으로 표시하는 방식이 더 실용적이다.
